# Fixer MCP GitHub Buildout

This directory is the clean publication track for a GitHub-ready Fixer MCP repository.

It exists because the current legacy workspace is effective for the original operator, but its public export still inherits private-layout assumptions that are awkward for external users.

## Glossary

- Fixer: the orchestrator that owns task dispatch, review, and canon updates
- Netrunner: the implementation worker that writes code, tests, and completion reports
- Overseer: the high-level coordinator that inspects the workspace and decides what kind of worker to start

## Goals

- replace the copy-and-strip export story with a first-class repository shape
- remove hardcoded personal workspace assumptions from the default onboarding path
- preserve backward compatibility for the legacy workspace and existing operators
- make follow-on implementation work divisible into safe, reviewable slices

## Legacy Assumptions Being Removed

- The old launcher bootstrap assumes a sibling `../mcp_servers` checkout or `MCP_SERVERS_ROOT` override.
- Public setup still describes the shared launcher runtime as an external prerequisite instead of packaging a clean default path.
- Repo-root config discovery (`mcp_config.json`, optional `webMCP.toml`) is tied to the private workspace layout.
- The current publication flow is generated by `scripts/prepare_public_repo.py`, which means the public repo is downstream from a private workspace instead of being the canonical source.
- Product docs still need to explain internal export mechanics to stay truthful.

## Architecture Summary

- `packages/fixer-mcp-server`: staged Go MCP server package with source, migration helpers, package-local examples, and validation commands
- `packages/client-wires`: publishable Python launcher package with a staged local runtime contract and compatibility fallbacks
- `packages/client-wires/config/`: package-local starter config used by the new track before any root-file discovery
- `packages/compat-bridge`: shims for legacy launch commands and migration helpers
- `examples/`: repo-level compatibility copy of the starter example for top-level discovery and docs
- `docs/`: public canon for architecture, migration, and implementation sequencing

## Public Bootstrap Quick Start

```bash
python3 -m pip install -e ./packages/client-wires
python3 -m fixer_client_wires wire-info
fixer-client-wires list-roles
fixer-client-wires plan-launch --role netrunner --backend codex --mcp-server fixer_mcp
```

The packaged `client-wires` track now prefers its staged local runtime and package-local `packages/client-wires/config/mcp-config.json`. `FIXER_CLIENT_WIRES_RUNTIME_ROOT`, `FIXER_CLIENT_WIRES_CONFIG_PATH`, and `FIXER_CLIENT_WIRES_CONFIG_ROOT` are the supported advanced overrides. `MCP_SERVERS_ROOT`, repo-root `mcp_config.json`, and sibling `../mcp_servers` remain migration-only compatibility paths.

Slice 4 adds a real staged launcher surface to `packages/client-wires`: packaged role definitions, backend descriptors, and a launch-plan CLI that previews headless launcher commands without importing the legacy `client_wires` workspace tree.

## Onboarding And Release

The public repo now has repo-native operator docs:

- [`docs/onboarding.md`](docs/onboarding.md) for first boot and role-oriented setup
- [`docs/release.md`](docs/release.md) for publication from `github_repo/` itself

The staged release helper lives at `scripts/release_public_repo.py`. It runs repo tests, builds the Python package artifacts through each package's declared PEP 517 backend, builds the staged Go binary, assembles the canonical public repo tree, and writes release metadata under `dist/releases/<version>/`.

The assembled publication boundary is now explicit:

```text
dist/
  releases/
    <version>/
      assembly/
        github_repo/
      bin/
        fixer_mcp
      python/
        client-wires/
        compat-bridge/
      assembly-manifest.json
      release-manifest.json
```

`dist/releases/<version>/assembly/github_repo/` is the first repo-native canonical source snapshot for publication review. It is generated from the tracked source tree inside `github_repo/` and no longer depends conceptually on the private export script.

## Incremental Migration Path

Operators moving from the legacy workspace can now install the staged compatibility bridge instead of translating everything by hand:

```bash
python3 -m pip install -e ./packages/client-wires
python3 -m pip install -e ./packages/compat-bridge
python3 -m fixer_compat_bridge --wire-info
fixer-compat-bridge --role fixer
```

`fixer_compat_bridge` accepts the old `--wire-info` and `--role` flag shape, keeps `MCP_SERVERS_ROOT` available as a compatibility override, and delegates into the GitHub-ready `client-wires` package. That makes the migration path executable without requiring edits to the legacy workspace outside `github_repo/`.

See [docs/architecture.md](docs/architecture.md) for the concrete design.
